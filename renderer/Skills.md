# Renderer Skills

## HTML Philosophy
- Render a document, not a product website.
- Default to a restrained corporate research tone.
- Prioritize readability, hierarchy, and print-like structure over interaction.
- Treat each report as a formal brief with clear sections and consistent spacing.

## Layout Rules
- Use a centered page layout that reads like a report on paper.
- Keep the content width narrow enough for long-form reading.
- Use a quiet background outside the document and a white document surface.
- Avoid dashboards, cards-as-homepage motifs, hero sections, marketing banners, and app-like navigation.

## Typography Rules
- Use a serif-led reading experience for body copy and a clean sans-serif for metadata.
- Keep heading hierarchy explicit and conservative.
- Prefer measured spacing and line length over visual novelty.
- Do not use oversized display text unless it serves document structure.

## Information Structure
- Present the report in a standard corporate research order:
  - Title / question
  - Executive summary
  - Introduction
  - Background
  - Analysis
  - Optional custom sections
  - Key findings
  - Confidence level
  - Limitations and assumptions
  - Suggested next steps
  - Sources
  - Diagnostics appendix when relevant
- Warnings and fallback states should appear as document notices, not as UI alerts.

## Visual Tone
- Use muted neutrals with restrained blue-gray accents.
- Keep borders thin and purposeful.
- Allow subtle page shadow and paper framing, but avoid decorative gradients and product-style visuals.

## Interaction Rules
- The first render is read-only.
- Editing will come later and should preserve the document-first presentation.
- Localhost delivery is acceptable, but the rendered page should still feel like a report prepared for review.
