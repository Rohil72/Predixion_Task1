# AI Research Agent Pipeline

An automated research system that utilizes multiple LLM agents (Planner, Search, Researcher, Formatter) to parse a complex query, search for information, synthesize it into a structured report, and render it locally. 

## Setup Instructions

1. **Prerequisites**: Ensure you have Python 3.10+ installed.
2. **Environment Variables**: The project requires API keys for OpenRouter (for LLM inference) and Tavily (for search).
   - Create a `.env` file in the root directory:
     ```env
     OPENROUTER_API_KEY=your_openrouter_api_key_here
     OPENROUTER_MODEL=openrouter/free
     TAVILY_API_KEY=your_tavily_api_key_here
     ```
3. **Execution**:
   - To run the pipeline orchestrator directly via Python:
     ```bash
     python agent_cli.py "Your research question here"
     ```
   - To run the end-to-end pipeline including rendering and server startup via PowerShell:
     ```powershell
     .\run_llm.ps1 "Your research question here"
     ```

## Architecture Overview

The system processes questions through a multi-agent pipeline:

1. **Planner (`planner_agent.py`)**: Decomposes the initial user prompt into a structured list of tasks (e.g., searches and synthesis), verifying its plan with the LLM or falling back to a deterministic model if required.
2. **Search (`search_agent.py`)**: Interacts with the Tavily search API to retrieve grounded context for the specific sub-tasks generated by the planner.
3. **Researcher (`research_agent.py`)**: Coordinates iterative model loops to synthesize evidence. It requests more data if needed and packages its findings into a comprehensive text payload.
4. **Formatter (`formatter_agent.py`)**: Takes the raw narrative output and structure from the researcher and maps it strictly into a schema-validated JSON format, applying diagnostics and extracting final citations.
5. **Renderer (`renderer/`)**: Parses the formatted JSON and generates an interactive, localized HTML report, which is then securely served on a local HTTP server (`renderer/serve_reports.py`).
6. **Guardrails (`guardrails.py`)**: Ensures strict input/output verification across all LLM inference points, protecting against prompt injection strategies and sensitive data leakage.

## Design Decisions

- **Single-Agent Nodes**: Rather than a monolithic ReAct agent, functionality is separated into specialized node processes that pass structured outputs to one another.
- **Strict JSON Contracts**: Interaction points (like the Formatter) validate payloads against a strict schema. Validations reduce formatting hallucinations and streamline rendering.
- **Stateless Execution**: Each sub-agent processes data largely statelessly via injected instructions instead of persisting global state, improving stability and making parallel execution and mocking easier for tests.
- **Simple Guardrails**: An efficient native implementation using blocklists and regex patterns performs guardrailing instead of a heavy external dependency (e.g. NeMo Guardrails or LLM-as-a-judge approaches) protecting the internal pipeline dynamically.

## Known Limitations

- **Context Window Exhaustion**: Current sequential searches append to the conversational context, which may overload strict token limits during rigorous factual explorations.
- **Stateless Rigid Pipeline**: The pipeline acts as a waterfall flow. The researcher cannot fundamentally challenge the initial planner's decisions if the topic proves contradictory to initial assumptions.
- **Search Budget**: There is a fixed budget to prevent runaway LLM costs, occasionally resulting in "fallback" reports.
- **Context Loss**: Intermediate insights gathered by early agents (e.g. Search Agent or Planner) might fade or summarize poorly since there is no persistent central memory to hold them globally across a single run.

## Future Improvements

1. **Larger Feedback Loop**: Allow downstream agents (like the researcher) to flag invalid assumptions back to the orchestrator to dynamically reshape the planning steps.
2. **Persistent Memory Agent**: Integrate a memory agent providing long-term and short-term conversational context storage across iterations, enabling cross-session continuation and better contextual retention.
3. **Task-based Model Switching**: Dynamically route queries based on complexity—using stronger, slower models (e.g., GPT-4o or Claude 3.5 Sonnet) for detailed synthesis while utilizing faster, cheaper models (e.g., Haiku or open-weight models) for simple metadata extraction or formatting.
4. **React-based Rendering and Editing**: Replace the static HTML generation with an interactive React frontend, allowing end-users much more interactivity such as inline report editing, manual source pruning, or iterative chat interactions embedded inside the report.
